# Руководство для контрибьюторов pagh

Спасибо, что хотите поучаствовать в `pagh` — небольшом 64-битном ядре на Rust
(`#![no_std]`, загрузка через Limine, запуск в QEMU/OVMF). Это хобби-/учебный
проект, поэтому здесь ценятся чистый код, чёткие коммиты и сохранение инвариантов
сборки. Документ описывает, что и как нужно сделать, чтобы ваш вклад приняли.

---

## 1. С чего начать

1. Прочитайте [`README.md`](README.md) целиком — там описаны архитектура, раскладка
   исходников (`src/`) и инварианты сборки. Без этого менять ядро рискованно.
2. Найдите или заведите issue под то, что хотите сделать. Для крупных изменений
   сначала обсудите подход в issue, чтобы не делать работу впустую.
3. Сделайте форк и заведите ветку от `main` (см. раздел про ветки и коммиты).

### Необходимое окружение

- **Rust nightly** с компонентом `rust-src` (нужен для `build-std`):
  ```sh
  rustup toolchain install nightly
  rustup component add rust-src --toolchain nightly
  ```
  `rust-lld` идёт в составе тулчейна и используется как линкер.
- **QEMU** (`qemu-system-x86_64`) и `qemu-img` в `PATH`.
- Два локальных блоба в корне проекта (они в `.gitignore`, их НЕ коммитят):
  - `OVMF.fd` — UEFI-прошивка для QEMU.
  - `limine-12.3.1/` — дерево загрузчика Limine (должно содержать `BOOTX64.EFI`).

`disk.img` создаётся автоматически при первом запуске — его тоже не коммитят.

---

## 2. Сборка и запуск

Вся сборка/линковка/запуск идёт через `run.cmd` (Windows):

```bat
run.cmd build           :: cargo build + линковка PAGH.elf
run.cmd run             :: build + линковка + загрузка в QEMU (по умолчанию)
run.cmd run release     :: release-сборка
```

Только статическая библиотека:

```sh
cargo build            # debug
cargo build --release  # release
```

Выход из QEMU: `Ctrl-A`, затем `X`.

## 3. Тесты — обязательны

Любое изменение логики должно быть покрыто и/или проверено тестами. В проекте два
уровня тестирования:

1. **In-QEMU self-test** (`src/test.rs`) — 27 свойств корректности ядра (P1–P27).
   Запуск из шелла командой `selftest`; результаты идут в serial как `ok`/`FAIL`.
2. **Host property-tests** (`host-tests/`) — `proptest` для логики, выносимой на хост
   (сейчас **40** модулей, `p01`–`p40`: ядро плюс Linux-ABI совместимости и конвейер
   `apt`/`deb`/`tar`). Это отдельный, исключённый из workspace крейт, собирающийся под
   хост-таргет:
   ```sh
   cd host-tests && cargo test
   ```
   Замечание: PowerShell может показывать stderr от cargo как `NativeCommandError` с
   ненулевым кодом выхода даже при успехе — ориентируйтесь на текст `test result: ok`,
   а не на код возврата.
3. **Boot-time Linux/apt self-tests** (`src/selftest_lx.rs`, cargo-фича `lx_selftest`) —
   сборка с `--features lx_selftest` запускает на старте проверки Linux-слоя syscalls,
   загрузчика статических ELF и конвейера `apt` против локального зеркала; они печатают
   `LXSELFTEST <name> PASS/FAIL` в serial. Повторяемые QEMU-харнессы лежат в `tools/`
   (`mini_repo.py` + `e2e_local_mirror.ps1`). **После сборки с фичей всегда пересоберите и
   верните в `iso_root/pagh.elf` дефолтное (без фич) ядро.**

Правила:
- Если меняете чистую логику (пути, line-editor, история, декодер, журнал, ext2,
  и т.п.) — добавьте/обновите соответствующее property-свойство.
- Если меняете Linux-ABI совместимости, конвейер пакетов (`deb`/`tar`/`apt_index`/
  `apt_resolve`) или сетевые парсеры (`dns`/`http`) — добавьте/обновите соответствующее
  свойство в `host-tests/` и, где это эффектный путь, boot-проверку в `lx_selftest`.
- Если затрагиваете аппаратно-зависимый путь — прогоните `selftest` в QEMU и убедитесь,
  что все строки `ok`.
- PR не должен ломать существующие свойства P1–P27.

> Не добавляйте тестовые фреймворки или зависимости ради одного теста — используйте уже
> принятые в проекте механизмы (`src/test.rs` и `host-tests/`).

## 4. Инварианты сборки (НЕ нарушать)

Эти ограничения зафиксированы по всему коду и должны сохраняться:

- `#![no_std]`, `panic = "abort"` (и в dev, и в release).
- Кастомный таргет `x86_64-unknown-none.json` с
  `build-std = [core, compiler_builtins, alloc]`.
- Статики запросов Limine лежат в секции `.requests`.
- Higher-half адрес загрузки `0xffffffff80000000` (`linker.ld`).
- Принудительно включённые frame pointers (`-Cforce-frame-pointers=yes`) — нужны для
  стек-трейса в панике.
- `host-tests/` НЕ должен становиться членом workspace ядра (он на хост-таргете).
- Код self-test / интеграции **закрыт фичами** (`lx_selftest`, `lx_livetest`); дефолтная
  сборка (без фич) должна оставаться неизменной по поведению, и именно дефолтный (без фич)
  ELF кладётся в `iso_root/pagh.elf`.

## 5. Стиль кода

- **Сборка без предупреждений.** Дерево собирается с нулём warnings — держите эту планку.
  Перед PR прогоните:
  ```sh
  cargo fmt --all
  cargo clippy
  ```
- **`unsafe` минимизируйте и документируйте.** Каждый блок `unsafe` сопровождается
  комментарием `// SAFETY:`, объясняющим, почему он корректен.
- **Привилегированные инструкции — только через `arch::cpu`.** Вне `arch` не должно быть
  инлайн-`asm!`, кроме неизбежных стабов в `task::switch` и перезагрузки сегментов GDT.
- **Никаких ссылок на `static mut`.** Глобальное изменяемое состояние достаётся через
  `SyncUnsafeCell`/атомики, чтобы не было `static_mut_refs`.
- **Разделяйте чистую логику и I/O.** Логику (нормализация путей, модель редактора,
  парсинг и т.п.) держите чистой и property-тестируемой; слой ввода-вывода — тонким.
- Следуйте существующей раскладке `src/` (см. README). Новый модуль кладите в подходящий
  каталог и подключайте идиоматично.

## 6. Ветки, коммиты и PR

- Ветка от `main`, осмысленное имя: `feat/<кратко>`, `fix/<кратко>`, `docs/<кратко>`.
- Коммиты — небольшие и атомарные, сообщение в повелительном наклонении и по сути
  («add ext2 dir iteration», а не «fixes»). Желателен формат Conventional Commits
  (`feat:`, `fix:`, `docs:`, `refactor:`, `test:`).
- Не коммитьте сгенерированные/локальные артефакты: `target/`, `host-tests/target/`,
  `iso_root/`, `PAGH.elf`, `disk.img`, `OVMF.fd`, `limine-12.3.1/`, логи QEMU, папки
  IDE. Всё это уже в `.gitignore` — не добавляйте их обходными путями.

### Чек-лист перед открытием PR

- [ ] `run.cmd build` (debug) проходит без ошибок и без warnings.
- [ ] `cargo fmt --all` и `cargo clippy` — чисто.
- [ ] `cd host-tests && cargo test` — зелёный (если затронута переносимая логика).
- [ ] `selftest` в QEMU проходит без `FAIL` (если затронут код ядра).
- [ ] Если собирали с `lx_selftest`/`lx_livetest` — дефолтное (без фич) ядро пересобрано и
      возвращено в `iso_root/pagh.elf`.
- [ ] Инварианты сборки из раздела 4 сохранены.
- [ ] Новый `unsafe` снабжён `// SAFETY:`.
- [ ] Документация (`README.md`, комментарии) обновлена, если менялось поведение.
- [ ] Нет лишних/сгенерированных файлов в diff.

### Описание PR

Кратко укажите: что и зачем меняется, как протестировано (host-тесты / `selftest` /
ручной прогон в QEMU), и ссылку на связанный issue. Для изменений в поведении — приложите
до/после (вывод serial, скриншот фреймбуфера и т.п.).

## 7. Сообщения об ошибках

Заводя issue, по возможности приложите:
- режим сборки (`debug`/`release`) и версию nightly (`rustc +nightly --version`);
- шаги воспроизведения и ожидаемое/фактическое поведение;
- релевантный вывод serial и/или фрагмент `qemu_debug.log`;
- для паники — стек-трейс (он печатается ядром по RBP-цепочке).

## 8. Порт на RISC-V (ветка `riscv-port`)

Экспериментальный порт на `riscv64gc` живёт в ветке `riscv-port` как отдельный
seed-крейт в `rv/` (собственный workspace, как `host-tests/`, поэтому он не влияет
на сборку x86_64-ядра). Цель — машина QEMU `virt`, загрузка через OpenSBI в S-mode.

- **Тулчейн:** тот же nightly + `rust-src`; seed собирается под
  `riscv64gc-unknown-none-elf` через `build-std` (настроено в `rv/.cargo/config.toml`).
- **Доп. инструмент:** `qemu-system-riscv64` в `PATH`.
- **Сборка и запуск:**
  ```bat
  run_rv.cmd            :: собрать seed и загрузить под QEMU virt + OpenSBI
  run_rv.cmd build      :: только собрать
  ```
  Временный образ `rv/rvdisk.img` создаётся при первом запуске (в `.gitignore`).
  Выход из QEMU: `Ctrl-A`, затем `X`.
- **План и статус:** карта переноса (x86_64→riscv64) и список этапов — в
  `.kiro/specs/riscv-port/`. Раздел «RISC-V port» в README перечисляет, что уже
  грузится (память, трапы/таймер, планировщик, U-mode + ecall, ns16550 UART,
  virtio-blk, virtio-net + DHCP, небольшой serial-shell).
- **Правило области:** архитектурно-независимая логика — общая; RISC-V-специфичен
  только слой архитектуры/платформы. Не ломать сборку x86_64 на `main`.
